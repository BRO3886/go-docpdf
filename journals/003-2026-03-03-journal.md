# Journal Entry 003 - 2026-03-03 - Migrate internal/metrics to prometheus/client_golang

## Goal

Replace the hand-rolled `sync/atomic` Prometheus renderer in `internal/metrics` with `github.com/prometheus/client_golang`, as specified in `docs/prds/prometheus-client.md` and GitHub issue #1.

## What Changed

- `internal/metrics/metrics.go` — complete rewrite (~100 lines → ~70 lines)
  - `prometheus.NewRegistry()` (custom, never default) + `MustRegister`
  - `CounterVec{outcome label}` replaces three separate `atomic.Int64` counters
  - `prometheus.Gauge` replaces `atomic.Int64` in-flight counter
  - `prometheus.Histogram` with same buckets `{100,250,500,1000,2500,5000,10000,30000}` replaces hand-rolled histogram
  - `promhttp.HandlerFor` replaces manual `fmt.Fprintf` renderer — gets content negotiation (text + protobuf) for free
  - All public method signatures unchanged (`IncSuccess`, `IncTimeout`, `IncFailed`, `IncInFlight`, `DecInFlight`, `ObserveDuration`, `ServeHTTP`)
- `internal/metrics/metrics_test.go` — tests rewritten to scrape text output via `ServeHTTP`; dropped `testutil.ToFloat64` approach (would have required exposing internal fields)
- `go.mod` — 1 direct dep added (`prometheus/client_golang v1.23.2`), 10 indirect
- `README.md` — removed "Zero external Go dependencies — stdlib only" from lede and updated package layout description
- `CLAUDE.md` — removed "No external dependencies — stdlib only" non-negotiable; added custom registry and pre-initialization notes

## Key Insights

**Zero-value counter series don't appear in client_golang exposition by default.** `CounterVec` only emits a label combination after it has been incremented at least once. The middleware test `TestMetrics_IncrementsSuccess` was asserting `outcome="failed" 0` and failing because the series was absent, not because the value was wrong. Fix: pre-initialize all three outcome values in `New()`:

```go
for _, outcome := range []string{"success", "timeout", "failed"} {
    conversions.WithLabelValues(outcome)
}
```

This makes zero-value series appear from the start, matching the hand-rolled renderer's behaviour.

**`TestContentType` needed relaxing.** The old test asserted the exact string `text/plain; version=0.0.4; charset=utf-8`. `promhttp` sets `text/plain; version=0.0.4; charset=utf-8` for plain text but the exact value can vary with content negotiation headers. Changed assertion to `strings.HasPrefix(ct, "text/plain")`.

**`go mod tidy` before writing code loses the dep.** Added the dep with `go get`, then wrote the new `metrics.go` importing the package, then ran `go mod tidy` to resolve properly. If you run tidy before writing the import, tidy removes the dep as unused.

## Decisions Made

- **Custom registry always** — `prometheus.NewRegistry()` not the default. Default auto-registers Go runtime and process collectors; those would change `/metrics` output unexpectedly and pollute tests.
- **Kept string-match test approach** — the PRD suggested using `testutil.ToFloat64` but that requires exposing internal metric fields. String matching against `ServeHTTP` output is simpler, tests the full rendering path, and requires no API surface changes.
- **No native histograms** — `NativeHistogramBucketFactor` left at 0 (disabled). Classic fixed-bucket histogram is sufficient; native requires Prometheus 2.40+ scraper.
- **Pre-initialize label values** — tradeoff: slightly opinionated (`New()` touches the counter), but necessary for behavioural parity with the old renderer and to avoid surprising missing-series bugs in production dashboards.

## Session 2 - Docker E2E validation

### Goal

Verify the prometheus/client_golang migration works end-to-end in Docker — zero-value series, counter increments, histogram bucket placement.

### What Changed

- `Dockerfile` — added `go.sum` to the `COPY` line alongside `go.mod`. Was `COPY go.mod ./`; now `COPY go.mod go.sum ./`. Without this, `go mod download` succeeds but the build fails with "missing go.sum entry" because Go 1.24 refuses to build without verified checksums. This would have broken any CI/CD build since the migration added real deps.

### Key Insights

**Dockerfile must copy go.sum when the module has external deps.** The old Dockerfile only copied `go.mod` because there were no external dependencies and go.sum was empty. After the prometheus migration, go.sum has 46 lines — omitting it causes `go build` to fail inside the builder stage even though `go mod download` passes.

**E2E metrics output confirmed correct:**
- Before any conversion: all three `outcome` counters present at 0 (pre-initialization working in Docker)
- After one conversion (1.94s): `success=1`, histogram landed in `le="2500"` bucket, `_sum=1940`, `_count=1`, `failed` and `timeout` still at 0
- In-flight gauge back to 0 after completion
